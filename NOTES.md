# PINN Project — Notes

## Що це

Physics-Informed Neural Network для 1D рівняння теплопровідності з рухомим пальником.

**Задача:** є металева труба довжиною l, по ній рухається пальник зі швидкістю v.
Знайти розподіл температури ΔT(x, t) для **будь-якого** матеріалу та швидкості — без
аналітичної формули під час навчання, тільки через фізику.

**Рівняння:**
```
∂ΔT/∂t = α · ∂²ΔT/∂x² + i_eff · δ(x − x₀ − v·t)
```
де:
- `α = k/(ρc)` — теплодифузійність [m²/s]
- `i_eff = intensity/(ρc)` — ефективна потужність пальника [K·m/s]
- `x_b(t) = x₀ + v·t` — позиція пальника в момент t

**Граничні умови (ізольовані кінці):** `∂ΔT/∂x = 0` при `x=0` та `x=l`
**Початкова умова:** `ΔT(x, 0) = 0`

**Мережа параметрична:** одна модель покриває весь простір матеріалів та швидкостей.
5 параметрів: α, l, i_eff, x₀, v — подаються як вхід разом з координатами (x, t).

## Архітектура мережі (повна)

### Вхід / вихід

```
Вхід:  (x*, t*)  ← безрозмірні координати ∈ [0,1]
       (Fo_n, x₀_n, β_n)  ← 3 безрозмірні π-групи, нормалізовані до [0,1]
Вихід: u  ← безрозмірний розв'язок (ΔT = T_c · u)
```

**Важливо:** мережа працює в безрозмірних змінних і виводить `u`, а не фізичний `ΔT`.
Фізична температура відновлюється поза мережею: `ΔT = T_c · u`, `T_c = i_eff·t_total/l`.
Деталі — у розділі «Знерозмірення» нижче. Це і є головний важіль, що пробиває стелю 68%.

### Fourier Feature Encoding

Перед подачею в мережу координати (x, t) проходять через **Random Fourier Features**:
```
φ(x,t) = [cos(2π·σ·B·[x,t]), sin(2π·σ·B·[x,t])]   ∈ R^128
```
де `B ∈ R^{64×2}` — фіксована матриця з N(0,1), `σ` — bandwidth що наростає під час навчання.

**Навіщо:** звичайний MLP страждає від spectral bias — він занадто повільно вчить
високочастотні деталі (вузький пік пальника). Fourier features явно дають мережі доступ
до потрібних частот через тригонометричні проекції.

### Adaptive Fourier Sigma (curriculum)

`σ` зростає лінійно від 1.0 до 3.5 протягом всього навчання:
```python
sigma = sigma_start + (sigma_end - sigma_start) * (step / total_steps)
```
**Навіщо:** на початку мережа вчить плавну глобальну динаміку (низький σ → низькі частоти).
Поступово σ збільшується і мережа отримує доступ до вищих частот для відтворення
вузького пальника. Це запобігає "перескакуванню" в локальний мінімум на ранніх кроках.

### PirateNet Blocks (Wang et al., JMLR 2024)

Після Fourier encoding + конкатенації з параметрами (133 → 256) іде 3 PirateNet блоки.

**Ключова ідея:** на початку навчання мережа поводиться як мілка (α≈0 → pass-through),
поступово "розблоковуючи" глибину через learnable residual gate.

```
[Один раз за forward pass, для всіх блоків:]
U = tanh(W_u · features)   ← shared encoder U (256-вимірний контекст)
V = tanh(W_v · features)   ← shared encoder V (256-вимірний контекст)

[Кожен PirateBlock(x_prev):]
h = tanh(W1 · x_prev)             ← звичайна трансформація
h = h⊙U + (1−h)⊙V                 ← gate 1: змішування через U,V
h = tanh(W2 · h)
h = h⊙U + (1−h)⊙V                 ← gate 2: знову змішування
h = tanh(W3 · h)                   ← фінальна трансформація (без gate)
out = α·h + (1−α)·x_prev           ← learnable residual skip (α init=0)
```

`α` — learnable scalar на блок, ініціалізований нулем. На початку: `out ≈ x_prev` (identity).
Поступово α зростає і блок починає вносити зміни. **α не обмежений** (no clamp) — так
прописано в оригінальному jaxpi коді.

**695 300 параметрів** (4 шари включно з input_layer, 3 PirateBlocks, encoder U/V, output).

### Hard Initial Condition

```python
output = t_norm * self.output_layer(h)   # замість просто output_layer(h)
```

При `t=0`: `output = 0 * NN(features) = 0` — IC виконується **математично**, а не через
штрафний терм. Це усуває цілий клас помилок де мережа "дрейфує" від IC на ранніх часах.
`lambda_ic = 0` в config — IC взагалі не треба оптимізувати.

## Знерозмірення — головний важіль (Buckingham-π)

### Проблема: масштаб ΔT охоплює ~3 порядки

Домінантний член розв'язку (n=0 мода ряду Фур'є) — це середній підйом температури:
```
ΔT_mean = i_eff · t / l
```
По всьому простору параметрів (`i_eff ∈ [1.25e-4, 5e-3]`, `t ∈ [7, 42]`, `l ∈ [0.5, 3]`):
```
ΔT_min ≈ 3e-4 K        ΔT_max ≈ 4e-1 K        розкид ~1400×
```

**Чому це вбивало навчання.** Loss — це сума квадратів PDE-residual по батчу з 16 наборів
параметрів. Residual має масштаб `i_eff/l`, тобто теж розкид ~1000×. «Гучні» набори
(великий i_eff / малий l) дають у ~10⁶ разів більший внесок у loss, ніж «тихі». Оптимізатор
фактично **ігнорує тихі набори** → їхня relative error ~100% → середнє по валідації витягується
у 68%. Це точно пояснює спостережувану стелю: ~половина наборів просто не навчені.

RAD/AMP/causal цього не лікують — вони про семплювання та швидкість, не про обумовленість.

### Другий баг: t_total не подавався в мережу

Старий вхід `[α_n, l_n, i_eff_n, x₀_n, v_n]` **не містив t_total**. Але нормований розв'язок
від нього залежить (через дифузію та позицію пальника). Два набори з різним t_total давали
мережі однаковий вхід, але різну правильну відповідь — мережа фізично не могла вгадати обидва.

### Рішення: знерозмірити задачу

Вводимо безрозмірні змінні:
```
x* = x/l        t* = t/t_total        u = ΔT/T_c        T_c = i_eff·t_total/l
```
PDE згортається до:
```
u_t* = Fo · u_x*x*  +  S(x*, t*)
```
де джерело `S = δ_gauss(x*; x_b*, σ*)`, `σ* = 1/50`, `x_b* = x₀_norm + β·t*`.

**Ключ:** пік `S ≈ 19.95` — **константа для всіх параметрів** (бо `σ_g = l/50` скорочується).
Отже residual O(1) скрізь → loss зважує всі набори однаково → мережа вчить весь простір.

### Три π-групи (теорема Бакінгема)

Безрозмірний розв'язок залежить рівно від трьох груп — замість п'яти параметрів:

| π-група | Формула | Зміст | Нормалізація |
|---|---|---|---|
| Fo | α·t_total/l² | число Фур'є (скільки дифузії) | log (4 порядки) |
| x₀_norm | x₀/l | старт пальника | лінійна [0, 0.3] |
| β | v·t_total/l | дистанція пробігу пальника | лінійна [0, 1] |

`Fo` нормалізується в **лог-просторі** (охоплює ~4 порядки: `[3.2e-6, 2.9e-2]`), інакше
малі значення стиснулися б у нуль. Це автоматично вирішує і проблему t_total (він всередині
Fo та β).

### Як це працює в коді

```
sampler.compute_pi_groups(raw)  → {Fo, x0_norm, beta, T_c}
model(coords*, pi_norm)         → u  (безрозмірне, O(1))
ΔT = T_c · u                    ← робиться у validate()/predictor, НЕ в мережі
```
PDE-loss рахується повністю в безрозмірних змінних (`physics.pde_loss`) — жодних фізичних
одиниць, residual завжди O(1). BC (`∂u/∂x*=0`) та hard IC (`u = t*·NN`) — без змін.

## Всі техніки навчання (повний список)

### 1. Causal PDE Loss (Wang et al., JMLR 2024)

**Проблема:** якщо ми оптимізуємо PDE на всіх часах одночасно — мережа може знайти
"псевдорішення" для великих t не знаючи правильної динаміки для малих t. Фізично:
щоб знати температуру в t=0.9, треба спочатку правильно вирахувати t=0.1.

**Рішення:** ділимо [0,1] на 10 часових вікон. Вікно k отримує вагу:
```
w_k = exp(−ε · Σ_{j<k} L_j)    де L_j = mean squared residual у вікні j
```
Поки `L_j` великий (рання динаміка погана) → `w_k ≈ 0` для пізніх вікон.
Мережа змушена вирішити задачу послідовно в часі.

**Параметри:** `causal_epsilon: 1.0`, `causal_n_bins: 10`

---

### 2. RAD — Residual-based Adaptive Distribution (arXiv:2207.10289)

**Проблема:** при рівномірному семплюванні більшість точок потрапляє туди де residual
вже малий — марна витрата обчислень. Для рухомого пальника residual сконцентрований
вузько навколо `x_b(t)` і рухається разом з ним. Рівномірна сітка погано це покриває.

**Рішення:** семплюємо набагато більше кандидатів, вибираємо де важко:
```
1. Рівномірно семплуємо pool_factor × n_pde = 5 × 8192 = 40 960 кандидатів
2. Обчислюємо |PDE residual| у кожному кандидаті (кінцеві різниці, no_grad)
3. Семплуємо n_pde = 8192 точок з ймовірністю ∝ |residual|
4. Кеш оновлюється кожні 50 кроків (між оновленнями — ті самі точки)
```
Точки автоматично концентруються навколо рухомого пальника без ручного налаштування.

**Чому кінцеві різниці, а не autograd для residual:**
RAD потребує тільки значення |residual|, не градієнти по вагах. З autograd + create_graph
на 40 960 точках — ~40× більше пам'яті. Кінцеві різниці (4 forward pass під no_grad)
дають достатню точність для оцінки "де погано" при мінімальних витратах.

```python
# Апроксимація d²ΔT/dx² і dΔT/dt через FD
dT_dxx ≈ (model(x+ε) - 2·model(x) + model(x-ε)) / ε²  / l²
dT_dt  ≈ (model(t+ε) - model(t)) / ε                   / t_total
residual = |dT_dt - α·dT_dxx - Q(x,t)|
```

**Параметри:** `rad_enabled: true`, `rad_pool_factor: 5`, `rad_update_every: 50`

### 3. AMP — Automatic Mixed Precision

**Проблема:** consumer GPU (RTX 5060 Ti) має FP32 продуктивність ~22 TFLOPS але
tensor cores оптимізовані під FP16 — матричні множення там у 2-4× швидші.
Чиста FP16 небезпечна для PINNs: при обчисленні похідних (d²ΔT/dx²) малі різниці
між великими числами можуть "зникнути" через вузький динамічний діапазон FP16.

**Рішення:** Automatic Mixed Precision (AMP) — гібридний підхід:
- Forward pass (матмули в моделі): **FP16** → швидко
- Gradient accumulation (оновлення ваг): **FP32** → точно
- GradScaler: масштабує loss вгору перед backward, масштабує градієнти вниз після.
  Якщо виявляє overflow (NaN/Inf) — зменшує scale вдвічі і пропускає крок.

```python
with torch.amp.autocast(device_type="cuda"):
    loss = total_loss(model, batch, ...)    # forward in FP16

scaler.scale(loss).backward()              # backward in FP32 (scaled)
scaler.unscale_(optimizer)                 # відновлює справжній масштаб градієнтів
clip_grad_norm_(...)                       # clipping після unscale
scaler.step(optimizer)                     # оновлює ваги
scaler.update()                            # адаптує scale factor
```

**Результат:** 9.3 it/s → **18.6 it/s** (2× speedup). 120k кроків: ~1.8 год замість ~3.6 год.

**NaN на перших ~15 кроках — нормально:** GradScaler стартує з великим scale,
виявляє overflow, ділить scale вдвічі, повторює. Після стабілізації NaN зникають.

**Поточний статус: вимкнено (`use_amp: false`).**
Нова безрозмірна постановка має джерело S з піком ≈ 19.95. Gradient через різкий Gaussian
(σ*=1/50) переповнює FP16 → GradScaler не може знайти стабільний scale → кожен крок NaN →
ваги заморожуються назавжди. Тренування просувалось по кроках, але жодного оновлення ваг
не відбувалось (~2000 steps = нуль навчання). AMP може повернутись якщо збільшити σ* або
перейти на серверний GPU із ширшим FP16-діапазоном.

**Параметр:** `use_amp: false`

### 4. FP64 — Double Precision (реалізовано, вимкнено)

**Джерело:** "FP64 is All You Need", NeurIPS 2025 (arXiv:2505.10949)

**Ідея:** FP32 може накопичувати помилки округлення у градієнтах, що призводить до
"stall phase" — loss зупиняється але помилка залишається великою.

**Чому вимкнено:** RTX 5060 Ti (як і більшість consumer GPU) має FP64:FP32 = 1:32.
Включення FP64 уповільнило тренування з 9.3 it/s до **1.7 it/s** (~5× повільніше),
ETA = 19 годин замість 2. На серверних GPU (A100: 1:2) FP64 виправданий.

**Параметр:** `use_fp64: false` — не вмикати на consumer GPU.

### 6. torch.compile (реалізовано, вимкнено)

**Ідея:** `torch.compile` аналізує граф обчислень і зливає CUDA-ядра у більші блоки,
зменшуючи overhead на запуск кожного ядра. Обіцяний speedup ~20-40%.

**Чому вимкнено — несумісність з `create_graph=True`:**

`torch.compile` (будь-який mode) використовує **AOT Autograd** (Ahead-of-Time), який компілює
backward з "donated buffers" — оптимізація де буфери входу backward передаються як буфери
виходу (memory aliasing). `pde_loss` викликає backward через скомпільовану модель з
`create_graph=True` (потрібно для ∂²u/∂x*² — grad of grad). AOT Autograd підіймає:
```
RuntimeError: This backward function was compiled with non-empty donated buffers
which requires create_graph=False and retain_graph=False.
```
Це фундаментальна несумісність у поточній версії PyTorch: donated buffers + create_graph=True
не можуть існувати разом. Правильний PyTorch 2.x fix — переписати `pde_loss` через
`torch.func.grad` + `torch.func.vmap` (functional transforms, compile-compatible), але це
серйозний рефакторинг. Зміна режиму (`mode="default"` замість `"reduce-overhead"`) не допомогла —
donated buffers є в усіх режимах compile.

**Додатковий конфлікт (CUDA Graphs у `reduce-overhead`):**
`mode="reduce-overhead"` додатково використовує CUDA Graphs, які фіксують один вхідний буфер.
При 4 різних викликах моделі в `pde_residuals_fd` (RAD) кожен наступний перезаписує буфер
попереднього. Зафіксовано через `.clone()` після кожного `model()` виклику.

**Параметр:** `use_compile: false`

### 7. Async Batch Prefetcher

**Проблема:** на RTX 5090 і RTX 5060 Ti однакова швидкість (~18 it/s). Причина — CPU
bottleneck: scipy LHS + Python-цикл генерації батча займали ~50 мс, GPU чекав.

**Рішення — два кроки:**

1. **scipy → `torch.rand`**: семплювання параметрів перенесено на GPU. LHS давав кращий
   per-batch coverage, але за 120k кроків різниця з random ≈ 0, а CPU↔GPU трансфер зник.

2. **`_BatchPrefetcher`** — фоновий thread генерує батчі, поки GPU рахує поточний крок:
```python
class _BatchPrefetcher:
    def __init__(self, cfg, normalizer, device, queue_size=4):
        self._queue = queue.Queue(maxsize=queue_size)
        self._thread = threading.Thread(target=self._worker, daemon=True)
        self._thread.start()

    def _worker(self):
        while not self._stop.is_set():
            self._queue.put(build_batch(self._cfg, self._norm, self._device))

    def get(self): return self._queue.get()  # main loop: нуль чекання
```
Main loop робить `prefetcher.get()` — бере готовий батч без затримки.
Коли RAD активний: PDE-точки замінюються на `rad_cache`, BC/IC беруться з prefetcher.

**Результат:** 18.6 it/s → **26+ it/s** на RTX 5070 Ti (разом з усуненням scipy).

**Параметр:** `use_prefetch: true` (вмикається автоматично на CUDA, вимикається на CPU/MPS)

### 5. Resume Fix — правильне відновлення LR при --resume

**Проблема:** `CosineAnnealingLR` в PyTorch використовує **рекурсивну** формулу:
```
lr_n = (1 + cos(π·n/T)) / (1 + cos(π·(n-1)/T)) · (lr_{n-1} − η_min) + η_min
```
Якщо при відновленні встановити `last_epoch=start_step-1` з `lr=5e-4` (initial),
планувальник зробить один крок і отримає `lr≈5e-4` (майже без змін).
Замість правильного `lr≈2.55e-4` для кроку 60k мережа отримала `lr≈4.4e-4` → overshoot.

**Рішення:** fast-forward планувальника від нуля:
```python
scheduler = CosineAnnealingLR(optimizer, T_max=adam_steps, eta_min=lr_min)
for _ in range(start_step):
    scheduler.step()   # застосовує рекурсивну формулу start_step разів → правильний LR
```
Це O(start_step) але кожна ітерація — тільки арифметика (мілісекунди).

## Валідаційна метрика

```
relative L2 error = ||ΔT_PINN − ΔT_analytical|| / ||ΔT_analytical||
```

Рахується для 8 випадкових наборів параметрів, у фіксований момент `t = 0.7 · t_total`.
Аналітичне рішення (`analytical_delta_T`) — ряд Фур'є (150 членів), використовується
**тільки для валідації**, не для навчання.

```
Baseline (MLP):                      76%
Запуск 3 (PirateNets):               68%   ← стеля масштабу
Запуск 4 (+ RAD + AMP):              ~68%  ← та сама стеля (масштаб не виправлений)

Запуск 5b (+ знерозмірення, без AMP): ???  ← ЗАРАЗ, очікуємо однозначні %
```

**5% без аналітичної формули.** Раніше це здавалось дуже амбіційним для параметричного PINN.
Тепер головна перешкода (обумовленість через масштаб ΔT) знята знерозміренням — задача
зведена до 3 добре обумовлених π-груп з O(1) виходом. Якщо 5% досяжні взагалі, то саме звідси.
Якщо після знерозмірення застрягне — наступні важелі: ProPINN (NTK-кореляція градієнтів)
або FP64 на серверному GPU.

## ProPINN — що це і чому важливо для проекту (не реалізовано)

**Джерело:** arXiv:2502.00803, лютий 2025.

**Root cause propagation failure:** доводить що PINN "провалюється" на часових задачах
через низьку кореляцію градієнтів між сусідніми точками по часу.

Математично через Neural Tangent Kernel (NTK):
```
K(t₁, t₂) = ⟨∇_θ f(t₁), ∇_θ f(t₂)⟩
```
Якщо `K(t, t+ε) ≈ 0` — оновлення ваг для точки t₁ майже не впливає на t₂.
Мережа не може "передати" знання вперед по часу — кожна точка вчиться ізольовано.

**Causal loss** (який у нас є) — це милиця проти симптому: примушує вчити ранні часи
першими. ProPINN вирішує причину: підбирає точки так щоб їхні градієнти були корельовані.

**Для фінального проекту:** ідеальна тема для розділу "чому PINNs важкі і що далі".
Покращення на конвекційних задачах (структурно схожих на рухомий пальник): +46%.
